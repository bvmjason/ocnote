# 🔄 如何验证与迭代爬虫

**让爬虫持续进化**

> 爬虫写好了不是结束，而是开始。持续验证、迭代优化，让爬虫越用越好用。

---

## 🤔 为什么要验证与迭代？

### 现实情况

**爬虫上线后会发生什么？**
```
第 1 天：爬虫运行正常，数据准确
第 2 天：爬虫运行正常，数据准确
第 3 天：网站改版了，爬虫抓取失败
第 4 天：发现数据有遗漏，但不影响使用
第 5 天：新增了一个字段，需要抓取
第 6 天：网站增加了反爬，需要调整
第 7 天：数据量增加，运行速度变慢
```

**结论：**
- 网站会变化
- 需求会变化
- 环境会变化

**所以：**
- 爬虫需要持续验证
- 需要不断迭代优化

---

## ✅ 数据验证方法

### 1. 数量验证

**方法：**
```
对比抓取数量与预期数量

示例：
- 网站显示：共 100 个商品
- 爬虫抓取：100 个商品
- 验证结果：✅ 数量正确

- 网站显示：共 100 个商品
- 爬虫抓取：95 个商品
- 验证结果：❌ 遗漏 5 个，需要检查
```

**实施：**
```python
# 验证代码示例
expected_count = 100  # 预期数量
actual_count = len(data)  # 实际抓取数量

if expected_count == actual_count:
    print("✅ 数量验证通过")
else:
    print(f"❌ 数量异常：预期{expected_count}，实际{actual_count}")
    print(f"差异：{expected_count - actual_count}")
```

---

### 2. 抽样验证

**方法：**
```
随机抽取部分数据，人工核对准确性

示例：
- 总共抓取 1000 条数据
- 随机抽取 20 条
- 人工核对每条数据的准确性
- 准确率 > 95% 则通过
```

**实施：**
```markdown
【抽样验证指令】

请帮我验证爬虫数据的准确性：

1. 从抓取的数据中随机抽取 20 条
2. 打开原始网页，人工核对以下字段：
   - 商品名称是否一致
   - 价格是否准确
   - 链接是否可访问

3. 统计准确率：
   - 20 条全部正确：✅ 通过
   - 18-19 条正确：⚠️ 需要检查
   - 少于 18 条正确：❌ 重新抓取

4. 记录错误类型：
   - 解析错误
   - 数据缺失
   - 格式错误
```

---

### 3. 对比验证

**方法：**
```
多源数据对比，发现异常

示例：
- 爬虫 A 抓取价格：¥100
- 爬虫 B 抓取价格：¥100
- 人工查看价格：¥100
- 验证结果：✅ 一致

- 爬虫 A 抓取价格：¥100
- 爬虫 B 抓取价格：¥120
- 人工查看价格：¥100
- 验证结果：❌ 爬虫 B 异常
```

**实施：**
```markdown
【对比验证指令】

请对比以下数据源：

1. 爬虫抓取的数据
2. 手动复制的数据
3. API 获取的数据（如果有）

对比字段：
- 商品名称
- 价格
- 库存状态

输出对比报告：
- 一致的字段
- 不一致的字段
- 可能的原因
```

---

### 4. 趋势验证

**方法：**
```
检查数据趋势是否合理

示例：
```
价格趋势：
- 昨天：¥100
- 今天：¥100
- 验证：✅ 正常（价格稳定）

- 昨天：¥100
- 今天：¥10000
- 验证：❌ 异常（价格暴涨，可能是解析错误）
```

销量趋势：
- 平时：每天 100 左右
- 今天：0
- 验证：❌ 异常（可能是抓取失败）
```

**实施：**
```markdown
【趋势验证指令】

请分析数据趋势：

1. 对比昨天和今天的数据
2. 检查以下指标：
   - 商品总数变化（±10% 内正常）
   - 平均价格变化（±20% 内正常）
   - 数据完整性（无大量缺失）

3. 标记异常：
   - 大幅波动（>50%）
   - 数据缺失（>20%）
   - 格式错误

4. 生成验证报告
```

## 🔄 迭代优化流程

### 标准迭代流程

```
第 1 步：发现问题
   ↓
   数据异常或功能失效
   
第 2 步：分析原因
   ↓
   定位问题根源
   
第 3 步：提出方案
   ↓
   制定修复方案
   
第 4 步：实施修复
   ↓
   修改代码或配置
   
第 5 步：验证结果
   ↓
   确认问题已解决
   
第 6 步：记录归档
   ↓
   记录问题和解决方案
```

---

### 常见迭代场景

#### 场景 1：网站改版

**问题：**
```
第 1 天：爬虫正常运行
第 2 天：抓取数据为空

原因：网站改版，HTML 结构变化
```

**迭代指令：**
```markdown
【迭代任务】网站改版适配

问题：
- 网站今天进行了改版
- 爬虫抓取数据为空
- 原选择器失效

新情况：
- 商品卡片 class 从 .product 改为 .item-card
- 价格 class 从 .price 改为 .current-price
- 新增字段：折扣信息

任务：
1. 分析新网页结构
2. 更新选择器配置
3. 添加新字段抓取
4. 验证数据准确性

请开始分析并修复。
```

---

#### 场景 2：数据质量优化

**问题：**
```
爬虫能运行，但数据质量不高：
- 商品名称包含多余空格
- 价格格式不统一
- 部分链接不完整
```

**迭代指令：**
```markdown
【迭代任务】数据质量优化

当前问题：
1. 商品名称包含多余空格和换行
2. 价格格式不统一（有的有¥，有的没有）
3. 部分链接是相对路径

优化要求：
1. 商品名称：
   - 去除前后空格
   - 去除多余换行
   - 统一空格为单个空格

2. 价格：
   - 统一去掉¥符号
   - 转为数字格式
   - 保留 2 位小数

3. 链接：
   - 补全为完整 URL
   - 验证链接可访问

请优化数据清洗逻辑。
```

---

#### 场景 3：性能优化

**问题：**
```
数据量增加后，爬虫运行变慢：
- 原来 100 条数据，运行 1 分钟
- 现在 1000 条数据，运行 30 分钟
```

**迭代指令：**
```markdown
【迭代任务】性能优化

当前情况：
- 数据量：100 → 1000 条
- 运行时间：1 分钟 → 30 分钟
- 瓶颈：每次请求间隔 2 秒

优化目标：
- 运行时间控制在 5 分钟内
- 保持数据准确性
- 不违反网站规则

优化方案：
1. 分析时间消耗分布
2. 优化请求策略
3. 添加并发处理（如允许）
4. 优化数据处理逻辑

请分析并提出优化方案。
```

---

#### 场景 4：功能扩展

**问题：**
```
业务需求变化，需要新增功能：
- 原来只抓取价格
- 现在需要抓取评论
- 需要分析情感倾向
```

**迭代指令：**
```markdown
【迭代任务】功能扩展

新需求：
1. 新增抓取商品评论
   - 评论数量（前 10 页）
   - 评论内容
   - 评论时间
   - 评分星级

2. 情感分析
   - 正面/负面/中性
   - 关键词提取
   - 情感得分

3. 输出扩展
   - 原 Excel 增加评论 sheet
   - 新增情感分析报告

请评估工作量并实施。
```

## ⚠️ 验证与迭代注意事项

### 注意事项 1：备份原数据

**做法：**
```
每次迭代前，备份当前数据

目录结构：
data/
├── backup/
│   ├── 2026-03-30_v1/
│   ├── 2026-03-30_v2/
│   └── 2026-03-30_v3/
└── current/
```

**指令：**
```markdown
在开始迭代前，请先：
1. 备份当前数据到 backup/2026-03-30_vX/
2. 记录当前版本号
3. 确认备份完成后再修改
```

---

### 注意事项 2：小步迭代

**错误做法：**
```
一次性修改多个地方：
- 改选择器
- 改数据清洗
- 改存储逻辑
- 改运行配置

结果：出问题不知道是哪里改的
```

**正确做法：**
```
每次只改一个地方：
第 1 次：只改选择器 → 验证
第 2 次：只改数据清洗 → 验证
第 3 次：只改存储逻辑 → 验证
第 4 次：只改运行配置 → 验证

每步都验证，问题容易定位
```

---

### 注意事项 3：记录变更

**做法：**
```markdown
每次迭代都记录：

变更内容：
- 改了哪里
- 为什么改
- 怎么改的

验证结果：
- 测试数据
- 验证结果
- 是否通过

回滚方案：
- 如果失败，如何恢复
- 备份位置
```

## 🎯 总结

**数据验证方法：**
1. 数量验证 - 对比预期与实际
2. 抽样验证 - 人工核对准确性
3. 对比验证 - 多源数据对比
4. 趋势验证 - 检查数据趋势

**迭代优化流程：**
1. 发现问题
2. 分析原因
3. 提出方案
4. 实施修复
5. 验证结果
6. 记录归档

**注意事项：**
1. 备份原数据
2. 小步迭代
3. 记录变更

**记住：**
> 爬虫不是一次性的，是持续进化的。
> 验证是常态，迭代是必须。
> 让爬虫越用越好用。
